# Assignment 02

## 1. Goals

### User Goals
1. **Analyze Performance Factors**: Understand how environmental variations (map layout, obstacles) and sensor disturbances influence robot localization and navigation performance.
2. **Identify Failures**: Automatically identify and classify failed runs or unusual behaviors in the simulation.
3. **Correlate Variables**: Find relationships between scenario parameters and performance outcomes.

### System and Feature Goals
1. **Automated Log Processing**: Ingest raw ROS 2 logs and CSV files to extract structured data.
2. **Metric Calculation**: Compute quantitative metrics for localization accuracy, heading accuracy, and navigation efficiency.
3. **Anomaly Detection**: Automatically flag runs that deviate from expected behavior based on defined thresholds and log patterns.
4. **Visualization**: Generate plots to visualize error distributions, anomaly occurrences, and correlations.

### Model Goals
1. **Accurate Classification**: Maximize recall to ensure all failed runs are identified.
2. **Predictive Capability**: Identify patterns in sensor data and logs that precede navigation failures.

## 2. Phenomena, Requirements, Specifications, and Assumptions

### World and Machine Phenomena

**World Phenomena ($W$)**:
- $W_{pose}$: The actual physical position and orientation of the robot in the simulation.
- $W_{env}$: The layout of the environment, including static obstacles.
- $W_{sensor}$: The physical readings generated by LiDAR and odometry sensors.
- $W_{collision}$: The physical event of the robot approaching or contacting an obstacle.

**Machine Phenomena ($M$)**:
- $M_{csv}$: The structured data rows in poses.csv (timestamps, pose values for `base_link` and `nav2_turtlebot4_base_link_gt`).
- $M_{log}$: The text strings recorded in ROS log files (e.g., "Goal succeeded", "Collision Ahead").
- $M_{metric}$: The calculated numerical values (e.g., RMSE, path length, time spent).
- $M_{alert}$: The output flag indicating an anomaly or failure.

### Behavioral Requirements (Functional)

| ID | Requirement |
|----|-------------|
| R_B1 | The system shall calculate the Euclidean error between the estimated pose (`base_link`) and ground truth pose (`nav2_turtlebot4_base_link_gt`) for every aligned timestamp. |
| R_B2 | The system shall classify a run as "Failure" if the log does not contain "Goal succeeded". |
| R_B3 | The system shall detect anomaly events by parsing log messages for keywords: "Collision Ahead", "spin failed", "backup failed", "Failed to make progress", "Failed to create a plan", "Goal failed", "FATAL". |
| R_B4 | The system shall compute run duration from "Begin navigating" to "Goal succeeded/failed" log timestamps. |
| R_B5 | The system shall calculate Mean Absolute Yaw Error normalized to [-π, π]. |

### Quality Requirements (Non-Functional)

| ID | Requirement |
|----|-------------|
| R_Q1 | **Reproducibility**: The pipeline must produce identical metric values given the same input dataset. |
| R_Q2 | **Interpretability**: Anomaly detection output must specify the anomaly type (e.g., "collision", "position_error"). |
| R_Q3 | **Modularity**: Code must be structured into reusable classes (`DataProcessed`, `LogsRos`) for future dataset expansions. |
| R_Q4 | **Robustness**: The system must handle missing CSV files or logs without crashing. |
| R_Q5 | **Performance**: The pipeline should process the complete dataset within reasonable time on a standard laptop. |

### Specifications ($S$)

| ID | Specification |
|----|---------------|
| S1 | When position error exceeds $Mean + 3\sigma$ for 3 consecutive frames, record a "position_error" anomaly. |
| S2 | If log contains keywords ("Goal failed", "spin failed", "FATAL", etc.), flag the specific anomaly type with timestamp. |
| S3 | Compute run duration as $T_{end} - T_{start}$ from log timestamps. |
| S4 | Synchronize poses by interpolating ground truth timestamps to match estimated pose timestamps using `np.interp`. |
| S5 | If run duration exceeds $Median + 3 \times MAD$ of the scenario, mark as "time_outlier". |

### Assumptions ($A$)

| ID | Assumption |
|----|------------|
| A1 | The `nav2_turtlebot4_base_link_gt` frame accurately reflects the true robot pose with negligible error. |
| A2 | The simulation environment produces comparable results across repeated runs with identical parameters. |
| A3 | Timestamps in CSV files are monotonically increasing and consistent across frames. |
| A4 | The "Goal succeeded" log message indicates the robot physically reached the target pose. |
| A5 | Absence of logs indicates a "no_initiation" failure where the robot never started. |

## 3. Performance Metrics and Data Preprocessing

### Data Preprocessing Rationale

1. **Timestamp Alignment**: The `base_link` (estimated) and `nav2_turtlebot4_base_link_gt` (ground truth) are published at different frequencies. Linear interpolation estimates ground truth at estimated pose timestamps.
2. **Missing Data Handling**: Runs without CSV files are flagged (`csv_file = None`) and excluded from metric calculations.
3. **Noise Filtering**: Position error anomalies require 3 consecutive threshold violations to filter transient sensor noise.
4. **Yaw Normalization**: Yaw error is normalized to [-π, π] using modulo arithmetic.

### Performance Metrics

| Metric | Formula | Rationale |
|--------|---------|-----------|
| **Euclidean Position Error** | $\sqrt{(x_{est} - x_{gt})^2 + (y_{est} - y_{gt})^2}$ | Standard metric for localization accuracy at each timestep. |
| **Mean Position Error** | $\frac{1}{N}\sum_{i=1}^{N} Error_i$ | Average localization performance over the run. |
| **RMSE Position** | $\sqrt{\frac{1}{N}\sum_{i=1}^{N} Error_i^2}$ | Penalizes large errors more heavily than mean. |
| **Max Position Error** | $\max(Error_i)$ | Captures worst-case localization deviation. |
| **Mean Absolute Yaw Error** | $\frac{1}{N}\sum_{i=1}^{N} \|\theta_{est} - \theta_{gt}\|$ | Measures heading accuracy, critical for path planning. |
| **Executed Path Length** | $\sum_{i=1}^{N-1}\sqrt{(x_{i+1}-x_i)^2 + (y_{i+1}-y_i)^2}$ | Total distance traveled; higher values may indicate inefficient navigation. |
| **Time Spent** | $T_{end} - T_{start}$ | Navigation efficiency; outliers indicate struggles. |

## 4. Anomaly Detection and Prediction

### Detection Methods

| Anomaly Type | Detection Method | Data Source | Single/Multi Source |
|--------------|------------------|-------------|---------------------|
| no_initiation | Rule-based | Logs (absence) | Single |
| fatal_initialization | Rule-based | Logs ("FATAL") | Single |
| planner_failure | Rule-based | Logs ("Failed to create a plan") | Single |
| no_progress | Rule-based | Logs ("Failed to make progress") | Single |
| collision | Rule-based | Logs ("Collision Ahead") | Single |
| stuck | Rule-based | Logs ("spin failed", "backup failed") | Single |
| goal_failure | Rule-based | Logs ("Goal failed") | Single |
| time_outlier | Statistical (MAD) | Run duration | Single |
| position_error | Statistical (3σ) | Pose CSV | Single |
| navigation_performance | ML (Isolation Forest) | Multiple metrics | Multi |

### Justification

**Rule-Based Detection**: Log keywords directly indicate system state changes. This approach is deterministic and interpretable.

**Statistical Detection**:
- **Time Outlier**: Uses Median Absolute Deviation (MAD) because sample size per scenario is small and MAD is robust against outliers. Threshold: $Duration > Median + 3 \times MAD$.
- **Position Error**: Uses global $Mean + 3\sigma$ threshold. Requires 3 consecutive violations to filter transient noise.

**ML-Based Detection**: Isolation Forest on combined metrics (mean_pos_error, rmse_pos, max_pos_error, mae_yaw, executed_path_length, time_spent) captures runs with unusual behavior patterns that may not trigger individual thresholds.

### Anomaly Relationships and Early Prediction

**Causal Chains Identified**:
1. **Collision → Stuck → No Progress**: Strong co-occurrence observed. Collision warnings are followed by recovery failures (stuck), leading to no progress.
2. **Accumulation → Time Outlier → Goal Failure**: Runs with repeated planner_failure or no_progress tend to become time_outliers before eventual goal_failure.

**Early Prediction Indicators**:
1. **Early Phase Error**: Higher mean position error in the first 30% of a run correlates with eventual failure (statistically significant via Mann-Whitney U test).
2. **Early Anomaly Count**: Anomalies occurring in the first 30% of run time predict total anomaly count and failure likelihood.

**Single vs Multi-Source Detection**:
- **Single-source**: All log-based anomalies and statistical anomalies (time_outlier, position_error) are detectable from one data source.
- **Multi-source**: navigation_performance anomaly requires combining multiple metrics, capturing holistic behavior patterns not visible in individual metrics.

### Key Findings

1. **Fatal Indicators**: "no_initiation" and "fatal_initialization" almost always result in immediate run failure.
2. **Accumulating Anomalies**: "planner_failure" and "no_progress" accumulate within problematic runs but are not always immediately fatal.
3. **Predictive Signals**: Early-phase localization errors and collision warnings serve as predictive signals for later failures.
